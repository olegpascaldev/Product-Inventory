🚀 Go Microservices: Product & Inventory
Этот проект представляет собой учебную реализацию двух микросервисов на Go с использованием gRPC, Kafka и PostgreSQL. Архитектура построена с учётом лучших практик: чистая архитектура, разделение на слои, интерфейсы, внедрение зависимостей, graceful shutdown, обработка ошибок и миграции. Код написан так, чтобы быть понятным, расширяемым и легко поддерживаемым.

Проект идеально подходит для изучения взаимодействия микросервисов, а также может служить основой для реальных приложений.

🧱 Общая архитектура
Система состоит из двух независимых сервисов:

Product Service — отвечает за управление товарами (создание, просмотр, обновление, удаление). При создании товара публикует событие ProductCreated в Kafka.

Inventory Service — управляет остатками товаров на складе. Подписан на Kafka и при получении события о новом товаре создаёт запись остатка с начальным количеством. Также предоставляет gRPC API для просмотра и изменения текущего остатка.

Взаимодействие
Синхронное через gRPC: Product Service может запросить у Inventory Service текущий остаток товара, чтобы показать полную информацию о товаре вместе с наличием на складе.

Асинхронное через Kafka: при создании товара Product Service отправляет событие, а Inventory Service его обрабатывает — это обеспечивает слабую связанность и надёжность.

Каждый сервис использует свою базу данных PostgreSQL, что соответствует паттерну Database per Service.

🛠 Технологический стек
Язык: Go 1.25+

База данных: PostgreSQL 14, библиотека pgx для работы с БД

Синхронный RPC: gRPC (протокол буферы) — для вызовов между сервисами

Асинхронная очередь: Apache Kafka (клиент sarama)

Контейнеризация: Docker, Docker Compose для локального запуска всех зависимостей

Управление зависимостями: Go Modules

Конфигурация: переменные окружения + .env файлы

Миграции БД: обычные SQL-скрипты (можно заменить на golang-migrate)

📁 Структура проекта
Общая структура репозитория
text
project/
├── docker-compose.yml          # оркестрация всех сервисов и инфраструктуры
├── product-service/             # первый микросервис
│   ├── cmd/server/              # точка входа
│   ├── internal/                # внутренний код (не для импорта извне)
│   ├── pkg/                      # сгенерированный protobuf код
│   ├── proto/                    # protobuf файлы
│   ├── go.mod
│   ├── Dockerfile
│   └── .env.example
└── inventory-service/            # второй микросервис
    ├── cmd/server/
    ├── internal/
    ├── pkg/
    ├── proto/
    ├── go.mod
    ├── Dockerfile
    └── .env.example
Что внутри каждого сервиса (на примере product-service)
cmd/server/main.go — инициализация всех зависимостей: подключение к БД, создание репозитория, usecase, Kafka-продюсера, gRPC-клиента для Inventory Service, запуск gRPC-сервера и graceful shutdown.

internal/config — загрузка конфигурации из переменных окружения.

internal/domain — чистые модели предметной области (Product, Stock и т.д.).

internal/repository/postgres — реализация интерфейса репозитория для работы с PostgreSQL.

internal/usecase — бизнес-логика: создание товара, получение с остатком, обновление, удаление. Здесь же определены интерфейсы для внешних зависимостей (репозиторий, Kafka, gRPC-клиент).

internal/delivery/grpc — gRPC-хендлеры, которые преобразуют запросы из protobuf в вызовы usecase.

internal/kafka — продюсер для отправки событий в Kafka.

internal/clients/inventory — gRPC-клиент для общения с Inventory Service.

pkg/protos — сгенерированный код из protobuf.

proto/ — исходные .proto файлы. В product-service лежат два файла: product.proto (свой API) и inventory.proto (чтобы выступать клиентом для Inventory Service). В inventory-service — только inventory.proto.

Такая структура следует принципам чистой архитектуры: зависимости направлены внутрь, бизнес-логика не зависит от внешних деталей.

🐳 Запуск проекта с Docker Compose
Требования
Docker и Docker Compose установлены.

Порты 5432, 5433, 9092, 2181, 50051, 50052 свободны (или изменены в конфигах).

Шаги
Клонируйте репозиторий.

Перейдите в корневую папку с docker-compose.yml.

Выполните команду:

bash
docker-compose up --build
Дождитесь запуска всех контейнеров: двух PostgreSQL, Zookeeper, Kafka, product-service, inventory-service.

Сервисы будут доступны:

Product Service gRPC порт: localhost:50051

Inventory Service gRPC порт: localhost:50052

Kafka: localhost:9092

PostgreSQL для product: localhost:5432

PostgreSQL для inventory: localhost:5433

Переменные окружения
Для каждого сервиса можно создать .env файл на основе .env.example. В docker-compose уже прописаны значения, подходящие для локального запуска.

🔍 Как пользоваться (примеры запросов)
Создать товар (через Product Service)
Используем любой gRPC-клиент (например, grpcurl или Postman).

bash
grpcurl -plaintext -d '{
  "name": "Ноутбук",
  "description": "Мощный игровой ноутбук",
  "price": 1500.00,
  "initial_stock": 10
}' localhost:50051 product.ProductService/CreateProduct
Ответ:

json
{
  "id": "uuid-товара",
  "name": "Ноутбук",
  "description": "Мощный игровой ноутбук",
  "price": 1500,
  "initial_stock": 10
}
Получить товар вместе с остатком
bash
grpcurl -plaintext -d '{"id": "uuid-товара"}' localhost:50051 product.ProductService/GetProductDetails
Ответ:

json
{
  "id": "uuid-товара",
  "name": "Ноутбук",
  "description": "Мощный игровой ноутбук",
  "price": 1500,
  "stock": 10
}
Обратите внимание, что поле stock пришло из Inventory Service по gRPC.

Изменить остаток (напрямую в Inventory Service)
bash
grpcurl -plaintext -d '{
  "product_id": "uuid-товара",
  "quantity_change": -2
}' localhost:50052 inventory.InventoryService/UpdateStock
Ответ:

json
{
  "product_id": "uuid-товара",
  "new_quantity": 8
}
Проверить остаток отдельно
bash
grpcurl -plaintext -d '{"product_id": "uuid-товара"}' localhost:50052 inventory.InventoryService/GetStock
Kafka события
При создании товара в логах контейнера inventory-service можно увидеть сообщение о полученном событии и создании записи остатка. Также можно зайти в контейнер Kafka и проверить топик product-events вручную.

🧪 Разработка и доработка
Генерация protobuf
Если вы меняете .proto файлы, нужно перегенерировать код. Для этого в каждом сервисе выполните:

bash
protoc --go_out=. --go-grpc_out=. proto/имя_файла.proto
При использовании Go modules важно указать правильный путь, чтобы импорты работали. В нашем случае в .proto указан option go_package = "product-service/pkg/protos/product"; (или аналогично), поэтому генерация будет корректной.

Миграции БД
Миграции лежат в папке migrations внутри каждого сервиса. При первом запуске Docker Compose автоматически создаст таблицы, если их нет. В реальном проекте можно использовать инструмент golang-migrate или встроить запуск миграций в код (но здесь для простоты это не реализовано).

Добавление новых методов
Добавьте описание метода в .proto.

Сгенерируйте код.

Реализуйте метод в gRPC-сервере.

Если метод требует бизнес-логики, добавьте её в usecase.

При необходимости расширьте репозиторий.

Обработка ошибок
Все ошибки оборачиваются в gRPC статусы с соответствующими кодами (NotFound, Internal, InvalidArgument и т.д.). Клиенты могут проверять код ошибки и реагировать.

💡 Ключевые архитектурные решения
Чистая архитектура позволяет легко тестировать и заменять компоненты (например, БД на in-memory хранилище).

Интерфейсы в usecase определяют зависимости, что делает код гибким.

Graceful shutdown обеспечивает корректное завершение работы: сервис перестаёт принимать новые запросы, дожидается завершения текущих и закрывает соединения.

Конфигурация через окружение — стандарт для двенадцатифакторных приложений.

Каждый сервис имеет свою БД — никакого прямого доступа к данным другого сервиса.

Асинхронность через Kafka развязывает сервисы и повышает отказоустойчивость.
